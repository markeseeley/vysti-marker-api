## A. Item 8 — `get_lexis_term` article-strip edge (`vysti_api.py` ~line 2242)

**Bug (verified).** `_normalize()` strips a leading `the/a/an ` from *every* query. When a
linked-term chip passes the **raw term text** (e.g. `"the Real"`, with a space), it is
normalized to `real` and resolves to the **wrong** entry. 13 lexicon terms carry a leading
article in `term_norm`; **4 actively mis-resolve to a different existing term**:

| raw term passed | OLD resolves to | correct |
|---|---|---|
| `the Real` | `real` ❌ | `the_real` |
| `the Symbolic` | `symbolic` ❌ | `the_symbolic` |
| `the Imaginary` | `imaginary` ❌ | `the_imaginary` |
| `the grotesque` | `grotesque` ❌ | `the_grotesque` |

The other 9 (`the gaze`, `a priori`, `the Apollonian`, `the male gaze`,
`the linguistic turn`, `the fantastic`, `the picturesque`, `the Pill`, `a fortiori`) fall
through to the fuzzy matcher (unreliable). The A-Z browse path (passes `term_norm` with an
underscore, no space) is unaffected — the regex requires a space, so only raw-text chips break.

**Fix.** Exact-match the *un-stripped* form first; only article-strip as a fallback.
Validated: resolves 13/13 article terms correctly, 0 regressions on non-article terms.

```diff

-        s = re.sub(r"['\"\[\]]+", "", s or "").strip().lower()
-        s = re.sub(r"^(the|a|an)\s+", "", s)
-        return re.sub(r"[^a-z0-9]+", "_", s).strip("_")
+        s = re.sub(r"['\"\[\]]+", "", s or "").strip().lower()
+        return re.sub(r"[^a-z0-9]+", "_", s).strip("_")
+
+    def _normalize_strip_article(s: str) -> str:
+        s = re.sub(r"['\"\[\]]+", "", s or "").strip().lower()
+        s = re.sub(r"^(the|a|an)\s+", "", s)
+        return re.sub(r"[^a-z0-9]+", "_", s).strip("_")

-    query = _normalize(term_norm)
-    if not query:
+    query_full = _normalize(term_norm)               # keeps "the Real" -> the_real
+    query = _normalize_strip_article(term_norm)      # article-stripped fallback
+    if not query and not query_full:
         return JSONResponse({"error": "Term not found"}, status_code=404)

     # Exact match first
-    matches = lexis_df[lexis_df["term_norm"].str.lower() == query]
+    matches = lexis_df[lexis_df["term_norm"].str.lower() == query_full]

+        matches = lexis_df[lexis_df["term_norm"].str.lower() == query]
```

(The remaining plural/substring/fuzzy fallbacks continue to use `query` unchanged.)

**Risk:** very low. Pure widening of exact-match before the lossy strip; no schema/data change.

## C. term_norm convention — 16 live rows don't match the stated formula

The project's mandated `term_norm` formula is
`re.sub(r"[^a-z0-9]+","_", term.lower()).strip("_")`. But **16 existing live rows** use a
*different, arguably better* convention: they **transliterate** accents and **keep hyphens**,
e.g.:

| term | stored term_norm | strict formula would give |
|---|---|---|
| `mise-en-scène` | `mise-en-scene` | `mise_en_scene` (and `mise_en_sc_ne` if not transliterated) |
| `Aufklärung` | `aufklarung` | `aufkl_rung` ← garbage |
| `akoumène` | `akoumene` | `akoum_ne` ← garbage |
| `amour-propre` | `amour-propre` | `amour_propre` |

The strict formula produces **broken keys for accented terms** (`sc_ne`, `aufkl_rung`) because
it turns each accented char into `_`. The data authors sensibly transliterated instead. So the
real issue is the **formula is too naive**, not the data.

**Verified:** none of these 16 are referenced by any other row's `linked_lexis`/`assign_lexis`,
so renaming would not break internal cross-references (only external bookmarks / raw-text chips).

**Recommended decision (yours):** adopt a refined canonical = *transliterate accents (e.g.
`unidecode`) → lower → non-alphanumerics to `_` → strip `_`*, and either (a) keep hyphens or
(b) collapse them to `_`. Then re-normalize the 16 rows (and `mise_en_sc_ne` from item B) in
one pass and **update the project directive** to match. This is a live lookup-key change → do
it deliberately with the `get_lexis_term` fuzzy matcher (already handles typos) as the safety net.

**Needs your call:** confirm the refined formula (hyphens kept vs collapsed) before any rename.
